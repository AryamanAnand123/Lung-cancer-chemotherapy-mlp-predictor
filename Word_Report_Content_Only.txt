CHAPTER 1
INTRODUCTION

1.1 Background
Lung cancer remains one of the highest-burden malignancies worldwide in terms of both incidence and mortality. Although treatment options have improved in the last decade, chemotherapy continues to play a central role in patient management, particularly for populations where targeted molecular options are unavailable, unaffordable, or clinically unsuitable. In such settings, reliable outcome prediction can provide practical support for clinicians by helping identify high-risk cases earlier, prioritizing follow-up intensity, and improving communication of risk with patients and caregivers.

Most real-world clinical pathways do not operate under ideal data conditions. Many hospitals and oncology centers, especially outside highly specialized institutions, do not maintain rich molecular panels, longitudinal symptom inventories, or deeply curated structured records. As a result, there is an operational gap between machine learning models that perform well in controlled publications and systems that can be deployed at scale in routine care environments.

This project is positioned in that gap. It focuses on a practical design principle: maximize predictive utility and interpretability while relying on broadly available registry-compatible variables and strongly auditable workflows. Rather than optimizing only for one headline metric, the project emphasizes discrimination, calibration, threshold behavior, and deployment continuity.

1.2 Project Overview
The project implements a dual-task machine learning framework for chemotherapy-related outcome modeling in lung cancer:
1) one-year mortality prediction, and
2) treatment-response prediction.

The modeling stack includes LightGBM, XGBoost, and TabNet. All model families are trained and compared using nested cross-validation, and all task-model runs generate a common reporting schema that includes fold metrics, outer means and standard deviations, confidence intervals, threshold optimization outputs, and visualization artifacts.

The framework is not limited to experimentation. It includes a serving layer that reads report metadata and auto-selects the best available model per task. This creates traceability from training evidence to production behavior and reduces the risk of undocumented manual model switching.

1.3 Motivation
The project is motivated by four practical needs:

First, reproducibility. Many studies present strong outcomes but do not provide report-level traceability that allows easy verification of claims.

Second, public-data feasibility. A significant number of clinical models rely on institution-specific features that reduce transferability.

Third, evaluation depth. AUC is necessary but not sufficient. Calibration and threshold behavior materially affect downstream usefulness.

Fourth, deployment realism. Offline evaluation and deployment selection are often disconnected, creating governance ambiguity.

This project addresses these needs by using a structured, artifact-backed pipeline where each numerical claim can be tied to generated files and each deployed model can be justified by report-ranked evidence.

1.4 Problem Statement
Current clinical machine learning pipelines for chemotherapy outcome prediction commonly face the following limitations:
1) dependency on non-portable proprietary variables,
2) insufficient transparency in model-selection process,
3) limited calibration and uncertainty reporting,
4) weak alignment between evaluation stage and deployment stage.

The problem addressed in this work is: how to design a dual-task predictive system that remains technically robust, practically deployable, and evidence-traceable when operating with registry-compatible feature spaces and heterogeneous source integration.

1.5 Objectives
The core objectives are:
1. Build an integrated dual-task architecture for mortality and response prediction.
2. Compare LightGBM, XGBoost, and TabNet in a consistent validation regime.
3. Evaluate performance using a broad metric set including AUC, F1, and Brier score.
4. Quantify uncertainty using fold-based confidence intervals.
5. Generate calibration, ROC, histogram, and SHAP artifacts.
6. Implement report-driven best-model auto-selection for deployment.
7. Produce a report structure suitable for academic review and reproducibility checks.


CHAPTER 2
LITERATURE REVIEW

2.1 Machine Learning for Lung Cancer Mortality and Survival Prediction
Recent literature has shown that machine learning can provide useful predictive signals for mortality and survival endpoints in lung cancer cohorts. Survival-centric studies often report C-index and time-dependent AUC values, while classification-oriented studies use binary horizon definitions such as one-year mortality labels. Both paradigms can be informative, but differences in endpoint definition complicate direct comparison.

A recurring concern in this literature is model-selection strategy. Some studies construct large candidate grids and select best-performing configurations using fixed test partitions, which may unintentionally introduce optimistic bias if not strictly isolated. This challenge is especially relevant when cohort sizes are modest and feature spaces are large.

Another challenge is feature portability. Many mortality pipelines incorporate detailed laboratory biomarkers, immunohistochemical markers, and patient-reported symptom trajectories. These features can improve performance but reduce deployment feasibility for institutions without the same data collection ecosystem.

Registry-based and mixed-source approaches have attempted to improve portability by emphasizing demographic and tumor-staging variables that are more consistently available. Such approaches typically trade feature richness for broader applicability, making robust validation and calibration reporting even more important.

2.2 Machine Learning for Chemotherapy Response and Complication Prediction
Response prediction and post-chemotherapy complication modeling have seen increasing use of classical ML models such as logistic regression, random forests, and gradient boosting. Results in this area are often strong when endpoint definitions are narrow and data quality is controlled, but variability rises when sample sizes are small or labels are heterogeneous.

A key issue in response literature is endpoint mismatch across studies. Some works predict treatment response classes, others predict chemotherapy-related infection risk, and others combine symptom burden with survival progression. Although these studies are related clinically, they are not directly equivalent statistically.

Class imbalance is another common challenge. Minority-class handling strategies (for example, resampling methods) can improve sensitivity but may distort calibration if not followed by strong reliability analysis. As a result, response-focused pipelines benefit from reporting both discrimination and calibration metrics, plus threshold sensitivity.

Advanced methods using molecular dynamics or mutation-specific pathways can produce very high performance in targeted cohorts, but those settings are often biologically narrow and difficult to generalize to broad mixed-population clinical workflows. This project therefore positions itself as a broad, practical dual-task system rather than a mutation-specific specialist model.

2.3 Explainability, Calibration, and Deployment in Clinical ML
The clinical ML field increasingly recognizes that useful models must satisfy more than a high AUC. Calibration quality affects whether output probabilities can be used for triage and risk communication. Threshold analysis affects operational decisions such as high-risk referral sensitivity. Explainability affects clinician trust, model auditability, and downstream acceptance.

SHAP has become one of the most widely adopted explanation frameworks for tabular tree models because it provides additive attribution with interpretable directionality and magnitude. While explanation does not prove causality, it helps assess whether model behavior is coherent with clinical intuition and data structure.

Deployment governance is another critical dimension. In many projects, the model selected in paper results differs from what is eventually served in production scripts, creating undocumented divergence. A report-driven selection policy can reduce this risk by formalizing deployment choices from reproducible evidence.

This project aligns with that modern perspective by integrating calibration, explainability, and deployment logic as mandatory outputs rather than optional post-hoc additions.

3.5 Model Selection and Training
Three model families were selected for complementary reasons:
- LightGBM: efficient tree boosting with strong tabular performance
- XGBoost: robust regularized boosting baseline
- TabNet: deep tabular attention architecture for comparative neural baseline

Validation protocol:
- Nested cross-validation with 3 outer folds
- Inner loop for hyperparameter search/selection
- Outer loop for unbiased performance estimation

Evaluation outputs include accuracy, balanced accuracy, precision, recall, F1, AUC, and Brier score at fold level and aggregate level.

CHAPTER 1 EXPANDED DISCUSSION (ADD AFTER SECTION 1.5)

Clinical decision-making in oncology is fundamentally probabilistic. Even when treatment protocols are standardized, patient-level outcomes vary due to age-related physiological reserve, tumor heterogeneity, disease spread, treatment adherence, and interaction effects that are difficult to represent in deterministic rules. Predictive analytics therefore serves as a decision-support layer rather than a replacement for clinician judgment. The practical value of prediction lies in stratification: identifying who is likely to require closer monitoring, earlier intervention, or intensified supportive care.

In routine care workflows, however, advanced AI systems fail not because algorithms are weak, but because data pipelines are not portable. A model that depends on institution-specific variables can perform very well where it is developed and fail when moved. This project deliberately takes a portability-first position by emphasizing variables that are commonly available in registry-style datasets. Such a strategy may not always produce absolute best-case experimental performance, but it produces systems that are easier to reproduce, audit, and adapt.

Another practical issue is communication of risk. Clinical users often ask, "How much confidence should we place in this probability?" AUC alone cannot answer this. Two models can share similar AUC and still differ in probability reliability and threshold utility. This is why the project includes calibration curves and Brier scores alongside discrimination metrics. In high-stakes domains, this multidimensional reporting is not optional; it is central to responsible deployment.

The project also addresses process continuity. In many academic pipelines, the model used in final deployment is selected manually from experiment logs and may not match manuscript claims exactly. This creates governance risk. The report-driven model selection logic in this work reduces ambiguity by making deployment follow ranked evidence from generated reports. The same artifacts that justify claims in Chapter 5 also determine operational model choice.


CHAPTER 2 EXPANDED CRITICAL REVIEW (ADD AFTER SECTION 2.3)

Methodological heterogeneity is a key reason why literature comparison in clinical ML is difficult. Papers differ in endpoint definition, cohort construction windows, exclusion criteria, censoring strategy, and missing-data treatment. For example, one-year mortality may be measured from diagnosis date, treatment initiation date, or first chemotherapy cycle date. Each design creates a different statistical problem. Therefore, when comparing reported AUC values across studies, endpoint alignment must be considered before claiming superiority.

Validation strategy is another major differentiator. A simple train-test split can produce inflated optimism in small or moderately sized cohorts. Nested cross-validation, while computationally more expensive, better separates model selection from final evaluation. This project uses nested CV not as a formal requirement but as a practical safeguard against accidental overfitting to a fixed partition.

Literature also shows a transition from pure discrimination reporting toward decision-aware modeling. Models that publish only AUC provide incomplete operational guidance because hospitals need threshold-level behavior. For instance, if sensitivity must be prioritized for high-risk referral pathways, threshold selection must be explicit and evidence-backed. This project includes threshold optimization outputs and documents best-threshold estimates per model-task combination.

Explainability literature highlights a persistent misconception: feature importance is often interpreted causally when it is only associational. The current project avoids causal claims. SHAP outputs are used to interpret model reliance patterns and check coherence with clinical expectations, not to infer biological mechanisms. This distinction is important for ethical reporting and for avoiding overstatement.

Finally, deployment studies emphasize maintenance. Predictive quality can drift due to data distribution changes, coding policy updates, and treatment-practice shifts. Although full drift-monitoring automation is outside the current scope, the artifact-first reporting design is compatible with periodic re-evaluation and model refresh cycles. This creates a practical pathway for future extension into post-deployment monitoring.


CHAPTER 3 IMPLEMENTATION DETAIL (ADD AFTER SECTION 3.8)

3.9 Data Harmonization Policy
Data harmonization was implemented to resolve schema variation across sources while preserving clinically meaningful fields. The harmonization policy had three priorities:
1) map equivalent semantics to common field names,
2) standardize encoding ranges and categories,
3) isolate source-specific fields that do not transfer cleanly.

Where multiple coding systems described the same concept, a canonical representation was selected and source-specific variants were transformed into that representation. This reduced feature ambiguity and simplified downstream model interpretation.

3.10 Missing Data Strategy
Missingness was handled in a task-aware manner rather than a single global rule. Features with extreme sparsity were either excluded or carefully imputed depending on endpoint relevance and available coverage. For tree models, structured missingness can sometimes be learned implicitly, but uncontrolled sparse features still increase variance and reduce interpretability. Therefore, sparse-field pruning was used to improve stability in both tasks.

3.11 Label Construction and Integrity Checks
Label integrity checks were included to avoid leakage and inconsistent endpoint assignment. For each task, records were filtered to ensure valid label presence and minimal contradictions. This step is critical because label noise can create unstable fold behavior and misleading confidence intervals.

3.12 Modeling Rationale by Family
LightGBM and XGBoost were selected as high-performance, interpretable tabular baselines with mature ecosystems and strong practical deployment history. TabNet was included to represent neural tabular modeling and assess whether deep attention mechanisms provide additive benefit in this setting.

The comparative framework intentionally keeps evaluation symmetric across model families. This is essential for fair ranking and for avoiding metric inflation due to asymmetric tuning depth.

3.13 Nested CV Configuration Notes
The outer-fold estimates were treated as the primary source for performance claims. Inner-loop optimization was used only for hyperparameter selection. This separation protects against optimistic leakage from tuning to final reporting.

3.14 Confidence Intervals and Stability Interpretation
Fold-based confidence intervals were calculated for AUC to express sampling variability in observed performance. While confidence intervals from few outer folds are still approximate, they provide a better uncertainty narrative than point estimates alone.

3.15 Threshold Optimization Procedure
Threshold optimization was carried out using task-specific objective behavior with emphasis on F1 balance. In practical deployments, threshold can be adjusted based on institutional policy priorities, for example prioritizing sensitivity for aggressive early escalation strategies.

3.16 Artifact Governance Design
Each run writes machine-readable reports and corresponding images into the reporting directory. The governance value of this design is that all major claims can be traced to files generated in the same execution flow. This reduces manual transcription error and strengthens reproducibility.

CHAPTER 5 EXTENDED ANALYSIS (ADD AFTER SECTION 5.5)

5.6 Comparative Interpretation of Mortality Results
The mortality task demonstrates a close competition between LightGBM and XGBoost. LightGBM slightly leads in AUC and Brier score, while F1 values are similar. This indicates that both models classify effectively, but LightGBM probabilities are marginally more reliable under Brier evaluation in the current run.

TabNet trails both tree models in mortality performance. This result is not unexpected in many tabular clinical datasets where gradient-boosted trees often provide stronger baseline reliability unless deep models are heavily tuned with larger feature spaces and larger data consistency.

5.7 Comparative Interpretation of Response Results
For response prediction, XGBoost clearly leads with the highest AUC and lowest Brier among tested models. LightGBM performs closely behind, suggesting that both tree ensembles capture useful predictive signal. TabNet performance on response is substantially weaker and less stable, likely due to limited sample size and endpoint complexity.

This finding supports a practical recommendation: for low-to-moderate sample tabular response tasks, tree boosting remains the preferred default unless strong justification exists for deep neural tabular alternatives.

5.8 Threshold and Decision-Policy Perspective
Default thresholds are often misaligned with operational goals. Optimized thresholds in this study provided measurable F1 improvements. In real clinical settings, threshold policy should be aligned to service objectives:
- sensitivity-prioritized threshold for early high-risk identification,
- precision-prioritized threshold for constrained intervention resources,
- balanced threshold when both false positives and false negatives carry substantial costs.

Because this project reports optimized thresholds and associated performance, institutions can adapt threshold policy without retraining the model.

5.9 Calibration Perspective
Calibration curves indicate how close predicted probabilities are to observed frequencies. Even when AUC is high, poor calibration can limit probability-driven decision support. The availability of calibration artifacts for all runs is therefore critical. It allows users to decide whether direct probabilities are trustworthy or whether post-hoc calibration adjustments are needed in future work.

5.10 Explainability Perspective
SHAP summaries help identify dominant feature contributions and validate that model behavior aligns with known clinical directionality trends. Explanations should be interpreted as model-attribution evidence, not causal evidence. In this report, SHAP is used to increase transparency and support qualitative model review.

5.11 Error and Risk Analysis
Potential risk factors in the current results include:
1) response-task sample limitations,
2) source heterogeneity residual effects,
3) endpoint-definition sensitivity,
4) potential institution-specific drift in external deployment.

These risks are mitigated in part by nested CV, confidence interval reporting, and artifact-backed diagnostics, but they are not fully eliminated. A responsible deployment strategy would include scheduled re-evaluation and periodic recalibration checks.

5.12 Practical Deployment Readiness Summary
From a practical standpoint, the project is deployment-ready for controlled pilot use:
- clear model ranking,
- documented thresholds,
- traceable artifacts,
- simple API serving flow.

For institutional rollout, additional governance layers are recommended, including formal version control of model bundles, audit logs of prediction requests, and periodic performance monitoring.
